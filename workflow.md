# プロジェクト開発ワークフロー（例：世界時計プロジェクト）

## 1. フォルダ構成・初期セットアップ
1.1 一般的なソフトウェアプロジェクトのフォルダ構成を作成する。
　- src/（ソースコード）
　- tests/（テストコード）
　- docs/（ドキュメント）
　- scripts/（スクリプト）
　- .github/（GitHub用設定）
1.2 必要に応じてサブフォルダ（modules, utils, config, unit, integration, design等）も作成する。

## 2. 仕様・要件定義
2.1 chater.md（プロジェクトの目的や背景、初期メモ）を作成する。
2.2 spec.md（仕様書案）を作成し、目的・機能・非機能要件・テスト方針を明確化する。
2.3 不明点や追加要件はCopilotに質問し、spec.mdへ随時反映する。

## 3. 開発環境構築
3.1 ローカルでWebアプリの動作確認ができるよう、簡易サーバ（Python http.serverやNode.js serve等）を用意する。
3.2 scripts/start_local_server.mdに起動手順を記載する。

## 4. 公開・運用方針
4.1 静的サイトとして構築し、公開用サーバはGitHub Pagesを利用する。
4.2 動的サイト構築は次回以降のプロジェクトで検討する。

## 5. Copilotとのやり取り
5.1 仕様や要件の明確化、設計・実装・テスト・運用の各フェーズでCopilotに相談し、内容をspec.mdやworkflow.mdに反映する。
5.2 無駄なやりとりや重複は省き、ワークフローの有効性を高める。

## 今回の開発で実施した主要な作業（サマリ）

- ドキュメント整理: `docs/` 配下に設計書・仕様・環境情報・検証レポートを配置（`docs/design/`, `docs/spec/`, `environment.md`, `docs/evaluation/validation1.md`）。
- 最小実装: `public/` に世界時計の最小UI（UTC / Asia/Tokyo の HH:MM:SS 表示）を追加、`src/js/clock.js` に時刻計算ロジックを実装。
- テスト導入: 単体テスト（`tests/unit/clock.test.js`）と Cypress を用いた E2E の枠組みを追加。簡易カバレッジ測定で `src/js/clock.js` の C0/C1 は 100% を確認（`tests/run_clock_runner.js` + c8）。
- 自動化: PowerShell スクリプトで Issue の一括作成・更新、`copilot` トリガーの GitHub Actions ワークフローを追加し自動実行を実装。
- CI と品質ゲート: `.github/workflows/ci.yml` を追加し、ユニット・E2E・スクリーンショット添付の必須化（`enforce-screenshots`）を導入。
- デプロイ: GitHub Pages に `gh-pages` ブランチで公開し、公開 URL を取得。
- ビジュアル検証準備: Playwright を導入し、`scripts/screenshot.js` を追加。ローカルでブラウザバイナリをダウンロードしスクリーンショット生成を確認したが、スクリーンショットは運用上は生成→削除または CI アーティファクトで管理する方針にしている。

## 学びと決定事項

- アセット参照は相対パスに統一する（`public/index.html` を修正）ことでローカルと Pages 表示の差異を解消した。
- `node_modules/` や Pages の生成物は `.gitignore` に追加して管理対象外とした。
- GitHub Actions からの push は権限設定で失敗する場合があるため、権限付与かローカルからの push のどちらかを運用ルールとして明確化する必要がある。

## 2週目（次スプリント）: 開発スコープ案（優先度順）

1. コア機能: アナログ時計表示の実装（Canvas/SVG または既存ライブラリ利用）と UI の動的管理（時計の追加/削除を 2〜8 台で可能にする）。
2. 都市選択 UX: 部分一致検索による候補表示と選択機能の追加（簡易 JSON データでスタート）。
3. 昼夜可視化: 文字盤の色で昼/夜を表示するロジックの実装。
4. i18n: `locales/` を用意して日本語／英語（拡張でタイ語／インドネシア語）を切替可能にする基盤実装。
5. テスト強化: `src/` の主要モジュールに対しユニットテストを拡充して C0/C1 を達成する（カバレッジ測定を CI に統合）。
6. ビジュアル回帰: CI 上でスクリーンショット取得 → 差分比較（例: Pixelmatch）を組み込み、PR の自動チェックを強化する。
7. ドキュメントとチケット: `spec.md` と `docs/evaluation/validation1.md` を元に各タスクを Issues に分解し、マイルストーンを設定する。

## 実行ルール（短く）

- PR: ロジック変更はユニットテスト（C0/C1）を含める。表示変更はスクリーンショットを添付する。CI がこれらを検証する。 
- ブランチ戦略: `main` は常にデプロイ可能状態。機能ごとにフィーチャーブランチを切る。 
- 自動化: `copilot` ラベルでトリガーされる自動ジョブはレビュー済みのテンプレートのみ実行する。
